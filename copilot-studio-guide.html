How It Works

The setup connects three things that already exist in your Microsoft environment. No new infrastructure, no third-party data processing.

Copilot Studio
your agent → Azure Functions
MCP server → Power BI API
your data model

Your Copilot Studio agent sends a question to the MCP server. The server translates it into a Power BI API call, runs the query, strips any personal data, and returns the result. The agent then formats it into a readable answer.

What is MCP? Model Context Protocol is an open standard that lets AI tools call external APIs in a structured way. Microsoft built native MCP support into Copilot Studio in late 2025. Think of it as a universal adapter between your AI agent and your data.

Available Tools

Once connected, the agent can use these seven tools:

ToolWhat it does
list_workspacesShows all Power BI workspaces the server has access to
list_datasetsLists datasets in a workspace
execute_daxRuns a DAX query against a dataset and returns the results
search_schemaSearches the data model for specific measures, columns, or tables
list_measuresLists all measure names in a dataset
list_fabric_itemsLists items in a Microsoft Fabric workspace
get_schemaReturns the full data model schema (use sparingly — can be large)

In a typical question, the agent chains 3–4 tools together: it discovers the schema, finds the right measures, builds a DAX query, and runs it. All automatically.

Deploy to Azure Functions 10 min

The deployment creates an Azure Function App in your subscription, uploads the server code, and configures it with your Power BI credentials. Everything stays within your Azure tenant.

# From the repo root cd azure ./deploy.sh

The script asks for four things interactively:

  1. A name for your Function App (e.g., acme-powerbi-mcp). Must be globally unique.
  2. Your Azure resource group (creates one if it doesn't exist, e.g., rg-powerbi-mcp)
  3. Your Power BI Workspace ID (the GUID from Step 1)
  4. Your Power BI Dataset ID (the GUID from Step 1)

The script handles everything else: creates the resource group, storage account, Function App, enables managed identity, configures environment variables, and deploys the code.

Platform notes:
macOS / Linux: Run directly in Terminal.
Windows: Use Git Bash (installed with Git), WSL, or PowerShell with bash deploy.sh. Alternatively, follow Option B which uses az commands that work natively in PowerShell and CMD.

For organizations that need to review each infrastructure change, here are the individual Azure CLI commands. Replace the placeholder values with your own.

1. Login and create resources

# Login to Azure az login # Create a resource group az group create \ --name rg-powerbi-mcp \ --location westeurope # Create a storage account (required by Azure Functions) az storage account create \ --name yourstorageaccount \ --resource-group rg-powerbi-mcp \ --location westeurope \ --sku Standard_LRS

2. Create the Function App

az functionapp create \ --name your-powerbi-mcp \ --resource-group rg-powerbi-mcp \ --storage-account yourstorageaccount \ --consumption-plan-location westeurope \ --runtime python \ --runtime-version 3.11 \ --functions-version 4 \ --os-type Linux

3. Enable managed identity

az functionapp identity assign \ --name your-powerbi-mcp \ --resource-group rg-powerbi-mcp

4. Configure environment variables

az functionapp config appsettings set \ --name your-powerbi-mcp \ --resource-group rg-powerbi-mcp \ --settings \ "AUTH_MODE=managed_identity" \ "POWERBI_MCP_WORKSPACE_ID=your-workspace-guid" \ "POWERBI_MCP_DATASET_ID=your-dataset-guid" \ "PORT=8000"

5. Deploy the code

# macOS / Linux — from the repo root cd azure zip -r deploy.zip host.json requirements.txt ../server/server_remote.py ../server/auth_remote.py # Windows (PowerShell) — from the repo root # Compress-Archive -Path azure\host.json, azure\requirements.txt, server\server_remote.py, server\auth_remote.py -DestinationPath azure\deploy.zip # Then deploy (all platforms) az functionapp deployment source config-zip \ --name your-powerbi-mcp \ --resource-group rg-powerbi-mcp \ --src deploy.zip

Verify the deployment

After deployment completes (either option), test that the server is running. The easiest way is to just open the URL in your browser:

# Open in your browser, or run in terminal: curl https://your-powerbi-mcp.azurewebsites.net/health # Windows (PowerShell): # Invoke-RestMethod https://your-powerbi-mcp.azurewebsites.net/health

Expected response:

{"status": "ok", "tools": 7, "auth_mode": "managed_identity"}
Save your URL. Your MCP server endpoint is https://your-app-name.azurewebsites.net/mcp — note the /mcp path. You need this in Step 5.

Configure Power BI Access 5 min

The MCP server needs permission to read your Power BI data. There are two authentication methods — Managed Identity is simpler and more secure.

Managed Identity is the simplest approach. Azure automatically handles the credentials — no secrets to store or rotate.

  1. Open the Azure Portal and navigate to your Function App
  2. Go to Settings > Identity > System assigned
  3. Verify the status is On (the deploy script already enabled this). Copy the Object (principal) ID.
  4. Open Power BI and navigate to your workspace
  5. Click Manage access (top right of the workspace)
  6. Click Add people or groups
  7. Paste the Object ID from step 3, select it, and assign the Member role
  8. Click Add
Why Member? The server needs to read workspace metadata, list datasets, and execute DAX queries. The Member role covers all of this without granting write access. Your data model, reports, and dashboards remain read-only.

Use a Service Principal if managed identity doesn't work for your setup — for example, if your Power BI workspace and Azure subscription are in different tenants.

  1. Go to Azure Portal > App registrations and click New registration
  2. Name it (e.g., PowerBI MCP Server), leave the default settings, click Register
  3. On the overview page, copy the Application (client) ID and Directory (tenant) ID
  4. Go to Certificates & secrets > New client secret, add a description, set expiry, click Add. Copy the secret value immediately (you can't see it again).
  5. Go to API permissions > Add a permission > Power BI Service > Delegated permissions
  6. Select Dataset.Read.All and Workspace.Read.All, click Add permissions
  7. Click Grant admin consent for [your org] (requires admin privileges)
  8. Add the app to your Power BI workspace as a Member (same as step 5–8 in the Managed Identity tab, but paste the Application ID instead)

Finally, update the environment variables on your Function App to use the service principal instead of managed identity:

az functionapp config appsettings set \ --name your-powerbi-mcp \ --resource-group rg-powerbi-mcp \ --settings \ "AUTH_MODE=service_principal" \ "AZURE_TENANT_ID=your-tenant-guid" \ "AZURE_CLIENT_ID=your-app-registration-client-id" \ "AZURE_CLIENT_SECRET=your-secret-value"
Secret rotation. Client secrets expire. Set a calendar reminder to rotate the secret before it expires, or use a certificate instead. Managed Identity (tab A) avoids this problem entirely.

Add the MCP Tool in Copilot Studio 5 min

Now that the server is running and has Power BI access, connect it to your Copilot Studio agent.

  1. Open copilotstudio.microsoft.com
  2. Select your agent (or create a new one — the default settings are fine for now)
  3. Click the Tools tab in the top navigation
  4. Click Add a tool
  5. In the search dialog, scroll down and find Model Context Protocol at the bottom, or type "MCP" in the search bar
  6. Click Model Context Protocol to open the configuration form
  7. Fill in the form:
    Server namePower BI
    DescriptionQueries Power BI datasets using DAX. Can list workspaces, datasets, measures, and execute analytical queries.
    Server URLhttps://your-app-name.azurewebsites.net/mcp (from Step 3)
    AuthenticationNone (the server authenticates with Power BI using its managed identity — no auth needed between Copilot and the server)
  8. Click Next. Copilot Studio connects to your server and discovers the available tools.
  9. You should see 7 tools listed. All toggles should be enabled by default. If not, enable them all.
  10. Click Add tool (or Save depending on your Copilot Studio version)
"No tools found"? Double-check the URL ends with /mcp (not just the Function App root). Also verify the health endpoint returns "tools": 7. If it returns "tools": 0, the server deployed but couldn't load its tool definitions — check the Function App logs.

The Connection Manager 2 min

This step is NOT documented in Microsoft's official MCP documentation. If you skip it, the tools will appear in your agent but silently fail on every query. This is the most common point where setups stall.

After adding the MCP tool, you need to activate the connection between Copilot Studio and your server. This is a one-time step that creates the actual runtime link.

  1. Go to the Test your agent panel (bottom-right in Copilot Studio)
  2. Type any question, for example: What workspaces do you have access to?
  3. The agent will attempt to use the Power BI tool and you'll see a message like: "The Power BI connection is not connected. Open the connection manager to fix this."
  4. Click the connection manager link in that message (or navigate manually to your agent's connection settings)
  5. You should see the "Power BI" connection listed with status Not Connected
  6. Click Connect
  7. If prompted, choose Create new connection and confirm
  8. Wait a few seconds. The status should change to Connected
You only do this once. After the connection is established, it persists across sessions. You don't need to reconnect unless you change the server URL or the connection expires.

Test Your Agent 2 min

Go back to the Test your agent panel and try these questions. Each one exercises a different part of the tool chain.

QuestionExpected tool chain
"What workspaces do you have access to?" list_workspaces → returns workspace names and IDs
"Show me all available measures" list_workspaces → list_datasets → list_measures → returns 200+ measure names
"What's our average CSAT score this month?" search_schema (finds the measure) → execute_dax (runs the query) → returns the value
"Top 10 clients by revenue this quarter" search_schema → execute_dax (with TOPN and SUMMARIZE) → formatted table
"Generate a live dashboard for SLA performance" search_schema → execute_dax (TOPN-limited) → DASH_CONFIG + DATA JSON objects
Token limit warning. Copilot Studio has an OpenAIModelTokenLimit that limits total output size. Always use TOPN(50, ...) in DAX queries to keep results within limits. Unlimited queries with hundreds of rows will trigger: "The request is resulting in too much data to handle."
Tip: be specific. "How are we doing?" is too vague. "What's our average ticket resolution time this quarter compared to last?" gives the agent something concrete to query. The more specific the question, the more accurate the DAX the agent writes.
If all four queries work, your setup is complete. The remaining steps are optional enhancements you can do later.

Customize Your Agent optional

Out of the box, the agent works but gives generic responses. You can make it more useful by adding context about your business.

Agent instructions

Go to your agent's Overview > Instructions and add context like this:

You are a Power BI assistant that helps users analyze business data using 7 MCP tools. TOOL USAGE ORDER — always follow this sequence: 1. list_workspaces — discover available workspaces 2. list_datasets — find datasets in a workspace 3. search_schema — find relevant tables, columns, measures (NEVER use get_schema — it crashes) 4. list_measures — list all measures 5. execute_dax — run DAX queries ANSWERING BUSINESS QUESTIONS: 1. Identify keywords from the question (revenue, tickets, hours, SLA, etc.) 2. Use search_schema to find matching measures and table names 3. Write DAX using the EXACT names found — never guess measure names 4. Execute the query and present results in clear tables 5. Show the DAX query you used 6. Suggest follow-up questions DAX GUIDELINES: - CRITICAL: Always use TOPN(50, ...) to limit rows — unlimited queries will crash - Use EVALUATE with SUMMARIZECOLUMNS or ADDCOLUMNS for grouped data - Filter dates: FILTER(ALL(DateTable[year]), DateTable[year] >= YEAR(TODAY()) - 1) - If a query fails, explain the error and try an alternative GENERATING DASHBOARDS (when user says "dashboard" or "live dashboard"): 1. Search relevant measures via search_schema 2. Pull data via DAX using TOPN(50) to limit rows 3. Output TWO JSON objects — DASH_CONFIG and DATA — nothing else 4. Use short field names in facts: rev, cost, hrs, t_cr, fr_pct, res_pct DATA JSON: {"dim":{"companies":["A","B"]},"facts":[{"c":0,"y":2025,"m":1,...}],"static_kpi":{}} DASH_CONFIG JSON: {"title":"...","kpi_rows":[[{...}]],"gauges":[{...}],"charts":[{...}],"table":{...}} Metric syntax: sum:field, ratio:a:b, wavg:sum:count, static:key OUTPUT RULES: - Never summarize or truncate data rows - For dashboards: output ONLY the two JSON objects, no HTML - The user will paste the JSON into a renderer page that builds the dashboard
Character count: These instructions use approximately 1,600 of the 8,000 character limit, leaving room for custom measure names and business-specific context.

Generate Live Dashboards optional

Your agent can generate interactive dashboards with KPI cards, gauges, charts, and company breakdowns. Due to Copilot Studio’s token limit, the agent outputs JSON configuration instead of full HTML. You paste the JSON into a Dashboard Renderer page that builds the visual dashboard.

How it works

  1. Ask for a dashboard: “Generate a live dashboard for SLA performance”
  2. Agent discovers data: calls search_schema to find relevant measures
  3. Agent pulls data: calls execute_dax with TOPN(50) to get company×month data
  4. Agent outputs JSON: two objects — DASH_CONFIG (layout) and DATA (facts)
  5. You paste into the renderer: open dashboard-renderer.html, paste the JSON, click Render
  6. Interactive dashboard appears: KPIs, gauges, Chart.js charts, filterable table
  7. Download: click Download HTML for a standalone file you can host or email

Dashboard Renderer

The renderer is a self-contained HTML page in the repository at templates/dashboard-renderer.html. Open it in any browser — no server required.

  • Split-screen layout: JSON input on the left, live dashboard preview on the right
  • Two textareas: one for DASH_CONFIG, one for DATA
  • “Paste from Copilot” button: auto-splits a combined JSON block into the two fields
  • “Download HTML” button: generates a standalone dashboard file with all CSS/JS embedded
  • Pre-populated sample: the renderer loads with sample data so you can see it working immediately

JSON structure reference

DATA uses a star-schema: dimension arrays (dim.companies) and fact rows with integer indices (c:0 = first company).

DASH_CONFIG defines the layout: kpi_rows (8 KPI cards), gauges (4–5 ring gauges), charts (4 Chart.js visualizations), and a table (sortable company breakdown).

Metric syntax: sum:field, ratio:a:b, wavg:sum:count, static:key. Formats: eur, pct, num, hrs, rating.

Why JSON instead of HTML? Copilot Studio enforces an OpenAIModelTokenLimit that caps total output. A full HTML dashboard (400–800 lines) exceeds this limit. JSON output (50–100 lines) fits comfortably, and the renderer handles the visual rendering client-side.

Production Hardening optional

For production deployments with multiple users, consider these additional steps.

API key authentication

By default, the MCP server accepts any request. To restrict access, add an API key:

# Set an API key on the Function App az functionapp config appsettings set \ --name your-powerbi-mcp \ --resource-group rg-powerbi-mcp \ --settings "MCP_API_KEY=your-random-api-key"

Then in Copilot Studio, update the MCP tool configuration to include the API key in the Authentication section (select "API Key" and paste the same key).

Monitoring

Azure Functions automatically logs to Application Insights. To enable it:

  1. Go to your Function App > Application Insights > Turn on
  2. This gives you query logs, error tracking, response times, and usage metrics
  3. Set up alerts for failures: Application Insights > Alerts > New alert rule > trigger on exceptions or failed requests

Scaling

The Consumption plan handles auto-scaling automatically. For consistent performance under heavy load, consider upgrading to a Premium plan (eliminates cold starts) or a Flex Consumption plan (pre-warmed instances). This is only needed if your team reports slow first-responses.

Network security

To restrict the Function App to only accept requests from Copilot Studio:

  1. Go to Function App > Networking > Access restrictions
  2. Add a rule to allow traffic from Microsoft's Copilot Studio IP ranges
  3. Set the default action to Deny
When to do this: If you're running a pilot with a few people, skip these steps. Come back when you're rolling out to the wider team or making the agent publicly available.

Troubleshooting

These are the issues we've encountered in real deployments, with proven fixes for each one.

"No tools found" when adding MCP server in Copilot Studio

Cause: The URL is wrong, or the server isn't running.

Fix: Open https://your-app.azurewebsites.net/health in a browser. It should return {"status": "ok", "tools": 7}.

  • If you get a 404: Make sure the URL is the Function App root (no /mcp for the health check, but /mcp for Copilot Studio)
  • If you get a 500: Check the Function App logs in Azure Portal > Functions > Monitor
  • If it doesn't load at all: The Function App might still be starting up (cold start). Wait 30 seconds and try again.
Tools appear but every query fails silently

Cause: You skipped Step 6 (Connection Manager). The tools are registered but the runtime connection isn't active.

Fix: Go back to Step 6. Test a query, click the "connection manager" link in the error message, and click Connect.

"Unauthorized" or 401 errors

Cause: The managed identity (or service principal) doesn't have Power BI access.

Fix: Verify three things:

  1. The identity was added to the Power BI workspace as a Member (not Viewer)
  2. "Allow service principals to use Power BI APIs" is enabled in the Power BI Admin Portal
  3. If using a service principal: admin consent was granted for the API permissions
"Forbidden" or 403 errors on execute_dax

Cause: The identity has workspace access but not dataset-level permissions.

Fix: In some configurations, the dataset has its own permission layer. Go to Power BI > dataset > Settings > and verify the identity has Build permission. Adding it as a workspace Member usually covers this, but if it doesn't, grant Build permission explicitly.

DAX queries return empty results

Cause: Either the dataset hasn't refreshed recently, or the measure names in the query don't match the model.

Fix: Ask the agent: "List all available measures." Compare the measure names with what the agent tried to query. Common issues:

  • Measure names with special characters or spaces not properly quoted
  • The dataset hasn't refreshed — check the last refresh time in Power BI
  • Filters are too restrictive (e.g., querying a date range with no data)
Copilot gives summarized or incomplete answers

Cause: The AI model compresses long outputs by default.

Fix: Be explicit in your question: "Show all rows, don't summarize." You can also add this to the agent instructions (Step 8): Always show the complete query results. Never summarize or truncate data tables.

Server timeout on large queries

Cause: The get_schema tool can return over 10MB of data, which exceeds the default timeout.

Fix: Always use search_schema instead of get_schema. Add this to the agent instructions: Never use get_schema. Always use search_schema to find specific measures or columns.

I see placeholder values like <PERSON> in the output

This is intentional. The server automatically detects and masks personal information (names, emails, phone numbers) before sending data to the AI model. This protects employee and client privacy.

The actual data in Power BI is unchanged. Only the data passed to the AI model is anonymized.

Connection manager shows "Expired" status

Cause: The connection token expired (usually after 90 days of inactivity).

Fix: Open the Connection Manager, click Refresh next to the Power BI connection, then Connect again. This takes about 10 seconds.

Function App shows "cold start" delays (10-15 seconds)

Cause: On the Consumption plan, Azure shuts down inactive functions after about 20 minutes.

Fix (free): Set up a health check ping. In the Azure Portal, go to Function App > Health check and set the path to /health. This keeps the app warm.

Fix (paid): Upgrade to a Premium or Flex Consumption plan for pre-warmed instances. Usually only needed if sub-second response times are critical.

Agent can't find the right measures for my question

Cause: The agent doesn't know your data model's naming conventions.

Fix: Add the key measure names to the agent instructions (Step 8). For example: Revenue is called "Revenue - Total". Utilization is called "Capacity Utilization Rate (Proxuma)". The more context the agent has about your naming conventions, the better it queries.

Deploy script fails with "subscription not found"

Cause: You're not logged in to the right Azure subscription.

Fix:

# List all subscriptions az account list --output table # Set the correct subscription az account set --subscription "Your Subscription Name" # Verify az account show --output table
"OpenAIModelTokenLimit" — too much data to handle

Cause: The DAX query returned too many rows. Copilot Studio enforces an OpenAIModelTokenLimit that caps the combined size of instructions + tool results + the agent's response. A query returning 100+ rows of company×month data easily exceeds this.

Fix: Wrap your query in TOPN(50, ...) to limit the result set. Add this to your agent instructions:

DAX GUIDELINES: Always use TOPN(50, ...) to limit results. Never query unlimited rows.

For dashboards, 50 rows is sufficient — the Dashboard Renderer handles client-side aggregation and filtering.

search_schema takes 10–15 seconds per call

This is normal. The search_schema tool scans the entire Power BI data model for matching measures and columns. Typical response time is 10–15 seconds per call.

Plan for it: A dashboard workflow calls search_schema 2–3 times (discovery phase), which takes 25–35 seconds total. After discovery, execute_dax queries are fast (1–5 seconds each).

Optimization: Add your most-used measure names directly in the agent instructions so the agent can skip discovery for common queries.

Company names show <ORGANIZATION> or <PERSON> in output

Cause: The MCP server has a PII (Personally Identifiable Information) anonymization filter enabled by default. It replaces detected company names, person names, and other identifiers with placeholders before the data reaches the AI model.

Impact on dashboards: If you use the Dashboard Renderer, the rendered charts and tables will show the anonymized placeholders instead of real names.

Fix: To disable PII anonymization, set ANONYMIZE_PII=false in the Function App environment variables (Azure Portal > Function App > Configuration). Only do this if your organization's data governance policy allows sending real names to the AI model.

Updated instructions don't take effect

Cause: Copilot Studio caches the previous instructions within an active test session.

Fix: After saving instruction changes, start a new test session by clicking the refresh icon at the top of the test panel. Changes only apply to new sessions, not to the one currently open.

Appendix: Environment Variables

Complete reference of all environment variables the MCP server supports.

VariableRequiredDefaultDescription
AUTH_MODE Yes — Authentication method. Either managed_identity or service_principal.
POWERBI_MCP_WORKSPACE_ID Yes — Default Power BI workspace GUID. Used when no workspace ID is specified in a query.
POWERBI_MCP_DATASET_ID Yes — Default Power BI dataset GUID. Used when no dataset ID is specified in a query.
PORT No 8000 Port the server listens on. Azure Functions sets this automatically.
MCP_API_KEY No — Optional API key. When set, all requests must include this key in the Authorization header.
AZURE_TENANT_ID SP only — Azure AD tenant GUID. Required when AUTH_MODE=service_principal.
AZURE_CLIENT_ID SP only — App registration client ID. Required when AUTH_MODE=service_principal.
AZURE_CLIENT_SECRET SP only — App registration client secret. Required when AUTH_MODE=service_principal.
PII_ANONYMIZE No true Whether to anonymize personal information in query results before sending to the AI model.
LOG_LEVEL No INFO Logging verbosity. Set to DEBUG for troubleshooting.
