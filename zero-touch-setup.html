What changed

From 10 Steps to 3

The old setup required cloning a repo, installing dependencies, finding GUIDs in Power BI URLs, editing a JSON config file, logging in, and verifying the connection. Ten steps. Fifteen minutes. Terminal expertise assumed.

The new setup runs a single command that installs everything, opens a browser for sign-in, auto-discovers workspaces and datasets, writes the config, and verifies the connection. Three steps the user sees. Two minutes total.

Enterprise-ready. IT admins can deploy to hundreds of machines with --silent mode, pre-configured IDs, config URLs, and environment variables. No user interaction required.

The auth module (auth.py) is shared between the wizard and the MCP server. The wizard's sign-in is also the server's sign-in — no double login. The auth record persists in ~/.powerbi-mcp/auth_record.json%USERPROFILE%\.powerbi-mcp\auth_record.json and tokens refresh silently for 90 days.

The wizard (wizard.py) calls the Power BI REST API to list workspaces and datasets. The user picks from numbered lists. The wizard writes the selected IDs into config.json. Enterprise deployments can bypass the wizard entirely with --workspace-id and --dataset-id flags.

How does the config.json look after the wizard runs?
~/.powerbi-mcp/config.json%USERPROFILE%\.powerbi-mcp\config.json
{ "default_workspace_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890", "default_dataset_id": "f9e8d7c6-b5a4-3210-fedc-ba0987654321", "workspaces": { "My Production Workspace": "a1b2c3d4-e5f6-7890-abcd-ef1234567890" }, "datasets": { "Sales Model": "f9e8d7c6-b5a4-3210-fedc-ba0987654321" } }

The wizard writes both the selected IDs and a friendly name mapping so you can see which workspace/dataset was chosen without decoding GUIDs.

The server (server.py) now supports environment variable overrides. IT can push POWERBI_MCP_WORKSPACE_ID and POWERBI_MCP_DATASET_ID via Intune, JAMF, or GPO — users don't touch config at all.

For IT admins

Enterprise Deployment

Deploy to hundreds of machines without any user interaction. Every flag works on both setup.shsetup.ps1.

Silent install with pre-configured IDs

terminal
curl -sL proxuma.io/install | bash -s -- \ --workspace-id "a1b2c3d4-..." \ --dataset-id "e5f6g7h8-..." \ --silent
PowerShell (Intune / SCCM)
.\setup.ps1 -WorkspaceId "a1b2c3d4-..." ` -DatasetId "e5f6g7h8-..." ` -Silent

Config from IT-hosted endpoint

Host a JSON config at a URL your users can reach. The wizard downloads it and skips the interactive picker.

terminal
./setup.sh --config-url https://it.yourcompany.com/powerbi-mcp-config
PowerShell
.\setup.ps1 -ConfigUrl "https://it.yourcompany.com/powerbi-mcp-config"

Device code flow (headless / SSH)

For environments without a browser (terminal servers, SSH sessions, Docker):

terminal
./setup.sh --device-code \ --workspace-id "a1b2c3d4-..." \ --dataset-id "e5f6g7h8-..."
PowerShell
.\setup.ps1 -DeviceCode ` -WorkspaceId "a1b2c3d4-..." ` -DatasetId "e5f6g7h8-..."

The user gets a URL and code to enter on any device with a browser. Authentication completes on the headless machine.

Environment variables (MDM / GPO)

Push config via Intune, JAMF, or GPO. These override any config.json on the machine.

VariableDescription
POWERBI_MCP_CONFIGFull JSON config blob. Overrides everything.
POWERBI_MCP_WORKSPACE_IDDefault workspace GUID. Overrides config.json.
POWERBI_MCP_DATASET_IDDefault dataset GUID. Overrides config.json.
Don't put secrets in env vars. These IDs are not secrets (they're workspace/dataset GUIDs), but never put passwords or tokens in environment variables. Auth tokens are handled by the Azure Identity library and stored securely in ~/.powerbi-mcp/%USERPROFILE%\.powerbi-mcp\.

Resolution order

Config Priority Chain

The MCP server resolves workspace and dataset IDs from multiple sources. Higher priority wins. This lets IT set defaults that users can override locally.

  1. POWERBI_MCP_CONFIG — Full JSON blob via environment variable. Overrides everything. Set via GPO, Intune, or JAMF.
  2. POWERBI_MCP_WORKSPACE_ID / POWERBI_MCP_DATASET_ID — Individual env vars. Override config file defaults.
  3. server/config.json — Local config in the repo directory. Checked first among file-based configs.
  4. ~/.powerbi-mcp/config.json%USERPROFILE%\.powerbi-mcp\config.json — Global config in user home. Written by the wizard. Fallback if no local config.
  5. Tool arguments — When the AI passes workspace_id or dataset_id directly to a tool call, those always win.
Enterprise pattern: Set POWERBI_MCP_WORKSPACE_ID and POWERBI_MCP_DATASET_ID via Intune/GPO. Users install with --silent. The server picks up the IDs from the environment. Zero touch.
What permissions does the server request?
ScopeWhat it does
analysis.windows.net/powerbi/api/.defaultRead Power BI data, execute DAX queries, list workspaces/datasets
api.fabric.microsoft.com/.defaultRead semantic model definitions (schema, measures, columns)

Both scopes are read-only. The server cannot modify your Power BI content. No Azure app registration needed — uses the same public client ID as Power BI Desktop.

How does PII protection work?

All data returned by MCP tools passes through Microsoft Presidio before reaching the AI. Personal information is automatically replaced with placeholders:

Detected PIIReplaced with
Person names<PERSON>
Email addresses<EMAIL>
Phone numbers<PHONE>
Credit card numbers<CREDIT_CARD>

The spaCy NLP model (en_core_web_sm) is downloaded during setup. Detection runs locally — no data sent anywhere.
