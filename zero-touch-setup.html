What changed

From 10 Steps to 3

The old setup required cloning a repo, installing dependencies, finding GUIDs in Power BI URLs, editing a JSON config file, logging in, and verifying the connection. Ten steps. Fifteen minutes. Terminal expertise assumed.

The new setup runs a single command that installs everything, opens a browser for sign-in, auto-discovers workspaces and datasets, writes the config, and verifies the connection. Three steps the user sees. Two minutes total.

Enterprise-ready. IT admins can deploy to hundreds of machines with --silent mode, pre-configured IDs, config URLs, and environment variables. No user interaction required.

The auth module (auth.py) is shared between the wizard and the MCP server. This means the wizard's sign-in is also the server's sign-in — no double login. The auth record persists in ~/.powerbi-mcp/auth_record.json and tokens refresh silently for 90 days.

The wizard (wizard.py) calls the Power BI REST API to list workspaces and datasets. The user picks from numbered lists. The wizard writes the selected IDs into config.json. Enterprise deployments can bypass the wizard entirely with --workspace-id and --dataset-id flags.

The server (server.py) now supports environment variable overrides. IT can push POWERBI_MCP_WORKSPACE_ID and POWERBI_MCP_DATASET_ID via Intune, JAMF, or GPO — users don't touch config at all.

For IT admins

Enterprise Deployment

Deploy to hundreds of machines without any user interaction. Every flag works on both macOS/Linux (setup.sh) and Windows (setup.ps1).

Silent install with pre-configured IDs

macOS / Linux
curl -sL proxuma.io/install | bash -s -- \ --workspace-id "a1b2c3d4-..." \ --dataset-id "e5f6g7h8-..." \ --silent
Windows (Intune / SCCM)
.\setup.ps1 -WorkspaceId "a1b2c3d4-..." ` -DatasetId "e5f6g7h8-..." ` -Silent

Config from IT-hosted endpoint

Host a JSON config at a URL your users can reach. The wizard downloads it and skips the interactive picker.

terminal
./setup.sh --config-url https://it.yourcompany.com/powerbi-mcp-config

Device code flow (headless / SSH)

For environments without a browser (terminal servers, SSH sessions, Docker):

terminal
./setup.sh --device-code \ --workspace-id "a1b2c3d4-..." \ --dataset-id "e5f6g7h8-..."

The user gets a URL and code to enter on any device with a browser. Authentication completes on the headless machine.

Resolution order

Config Priority Chain

The MCP server resolves workspace and dataset IDs from multiple sources. Higher priority wins. This lets IT set defaults that users can override locally.

  1. POWERBI_MCP_CONFIG — Full JSON blob via environment variable. Overrides everything. Set via GPO, Intune, or JAMF.
  2. POWERBI_MCP_WORKSPACE_ID / POWERBI_MCP_DATASET_ID — Individual env vars. Override config file defaults.
  3. server/config.json — Local config in the repo directory. Checked first among file-based configs.
  4. ~/.powerbi-mcp/config.json — Global config in user home. Written by the wizard. Fallback if no local config.
  5. Tool arguments — When the AI passes workspace_id or dataset_id directly to a tool call, those always win.
Enterprise pattern: Set POWERBI_MCP_WORKSPACE_ID and POWERBI_MCP_DATASET_ID via Intune/GPO. Users install with --silent. The server picks up the IDs from the environment. No config file editing, no wizard interaction. Zero touch.
